**Fundamental Principle: Component Singularity & Junction Role**
- Each component ID (e.g., `mixer_1`, `inlet_5`) refers to **ONE** single physical component instance on the chip with defined input/output ports (typically one input, one output, except for Filters/Droplet Gens).
- The `connections` array lists **conceptual** desired flows between these components.
- A single component ID can appear multiple times as a source or target in the conceptual `connections` list. This indicates a need for flow branching or merging.
- **CRITICAL:** **DO NOT** invent different "paths," "modes," or "versions" for a single component ID (like `mixer_1`). The physical routing required to handle multiple conceptual inputs or outputs for a single component is achieved **SOLELY** through the **binary junctions** detailed in the `junctions` array, which connect physically to the component's single input or output port.

**Component Definitions & Naming:**

- **Inlet (`inlet_X`):** Source. 1 output port (`inlet_X`).
- **Outlet (`outlet_X`):** Exit. 1 input port (`outlet_X`).
- **Mixer (`mixer_X`):** 1 input (`mixer_X`), 1 output (`mixer_X`). Param: `num_turnings` (default 4).
- **Delay (`delay_X`):** 1 input (`delay_X`), 1 output (`delay_X`). Param: `num_turnings` (default 4).
- **Chamber (`chamber_X`):** 1 input (`chamber_X`), 1 output (`chamber_X`). Params: `length` (default 4000), `width` (default 3200). Units are µm (implied).
- **Filter (`filter_X`):** 1 input (`filter_X`), 2 outputs (`filter_X_smaller`, `filter_X_larger`). Param: `critical_particle_diameter` (default 10). Units are µm (implied). Both outputs MUST be connected conceptually; connect to a new waste `outlet_X` if destination unspecified.

    - Output `droplet_X`: Outputs the generated droplets.

      3. Document this automatic addition in the reasoning.

**Junction Rules (CRITICAL):**
- Junctions (`junction_X`) are detailed in the `junctions` array. Their sole purpose is to provide the physical implementation (using **binary** splits or **binary** combines) necessary to achieve the **Conceptual Component-Level Flow** defined in the `connections` array.
- **Goal:** Define the **absolute minimum number** of necessary binary junctions required to correctly route flow according to the conceptual connections. Structure these junctions efficiently.
- **When Required:** Junctions are required *only* at points where implementing the conceptual connections demands a single physical component port handle connections from multiple distinct conceptual sources or to multiple distinct conceptual targets. (This is determined by analyzing the component connectivity summary in Reasoning Step 4).
- **Handling Complex Flows & Nesting:** Multi-way splits (>2 targets) or combines (>2 sources) identified *must* be implemented in the `junctions` array by **connecting binary junctions sequentially**. Nesting junctions is expected and correct.
    * *Example:* If conceptual connections (Step 2b) show `inlet_1->mixer_1`, `inlet_2->mixer_1` AND `mixer_1->outlet_1`, `mixer_1->outlet_2`, the analysis (Step 4) will identify `mixer_1` needs combining at input and splitting at output. The `junctions` array (Step 5) would then detail the necessary binary junctions to achieve this.
- **Numbering & Type:** Number junctions sequentially (`junction_1`, ...). Assign `Y-junction` or `T-junction` (use prompt hints if provided, otherwise default).
